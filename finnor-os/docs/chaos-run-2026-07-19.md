## A real bug found and fixed while running it

The first run failed during cleanup with a foreign-key violation:
`update or delete on table "workflow_steps" violates foreign key constraint
"decision_receipts_workflow_step_id_fkey"`. Cause: `decision_receipts` (added Phase 2,
migration `0016_decision_receipts_and_dlq.sql`) references `workflow_steps`, but
`chaos-test.ts`'s cleanup function predates that table and never deletes receipts
before deleting the steps they reference — the same class of pre-existing-test-cleanup
bug Phase 2's Task 2.4 already found and fixed twice in other test files
(`vertical-workflows-phase4.test.ts`, `workflow-runtime.test.ts`). Fixed in
`scripts/chaos-test.ts`'s `cleanup()`: deletes `decision_receipts` for each step id
before deleting `workflow_steps`, matching this repo's own established FK-order
convention. Verified stable: re-ran the full 3-scenario suite afterward, all 3 passed,
cleanup left zero orphaned rows (`workflow_runs` count for the chaos tenant confirmed 0
after the run).

## Real results, this session, local Postgres, real process kills

| # | Scenario | Kill point | Verdict |
|---|----------|-----------|---------|
| 1 | Pre-commit kill (before the step claim UPDATE ever runs) | `pre_commit` | **EXACTLY-ONCE** — step completed, single `integration_operations` row |
| 2 | Post-commit-pre-ack kill (real effect happened, crash before recording it) | `post_commit_pre_ack` | **RECONCILIATION_CASE OPENED** — correct, honest behavior for a genuinely unknown-delivery outcome; never silently retried or silently lost |
| 3 | Mid-multi-step kill (step 1 committed, crash before step 2 is enqueued) | `mid_multi_step` | **EXACTLY-ONCE** — step 1 completed once, step 2 recovered and completed by the fresh process |

All three scenarios spawn a real, separate OS process to run the step (which
self-kills via `SIGKILL` at the specified hook — confirmed via exit code 137 in every
scenario, i.e. the kill genuinely fired), then a second, completely fresh process (no
shared memory) recovers via `recoverStaleSteps` + drives the workflow to completion —
exactly what a real worker restart after a crash would do. Assertions are against
durable Postgres state only (`workflow_steps`, `integration_operations`,
`reconciliation_cases`), never the emulators' in-memory state, which correctly does not
survive the kill (matching how a real external provider's state lives outside this
process's memory).
