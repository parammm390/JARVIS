# Load test — 2026-07-19 (Phase 6, Task 6.4)

## What's real here

`scripts/k6-load-test.js` — a real k6 script matching the pack's exact scenario (50
inbox events/s sustained 10 min via `POST /api/webhooks/marketing`, 200 concurrent
read-model queries across all 9 views including the new `reliability` one, 20 approval
round-trips/min via `GET /api/actions/pending` + `POST /api/actions/:id/confirm`),
written against the real route contracts, not guessed:

- Read every route it targets before writing the script (`webhooks/marketing/route.ts`,
  `read-models/[view]/route.ts`, `actions/pending/route.ts`, `actions/[id]/confirm/
  route.ts`) to get request/response shapes exactly right.
- Smoke-tested the underlying HTTP layer for real, this session, against a locally
  running `apps/api` dev server (real Next.js route handlers, real embedded Postgres,
  real Dealer Zero data) — not a guess:
  - `GET /api/read-models/reliability` → 200, real computed metrics (workflowSuccessRate:
    1, stepLatencyMs sampleSize 1056, humanInterventionRate ~0.054, etc.)
  - `POST /api/webhooks/marketing` with a shared secret → 200, real lead + household
    rows created (`leadId`, `householdId` returned)
  - `GET /api/actions/pending` → 200, real pending `create_invoice` actions from Dealer
    Zero's live simulator/detection loops
  - Along the way, confirmed a cross-tenant-looking result from this smoke test was the
    **already-documented local-dev-only RLS bypass** (the embedded Postgres dev role is
    effectively superuser, so routes relying on RLS alone — no explicit tenant filter —
    leak across tenants locally but not in production; `tests/integration/tenant-
    isolation.test.ts` already proves the exact same code path is tenant-isolated under
    the real non-superuser `finnor_app` role). Not a new finding — verified against the
    existing test before concluding anything, per this repo's own established caution
    around that specific gotcha.

## What's NOT run yet, stated plainly

**k6 itself is not installed in this development environment** (`which k6` finds
nothing, confirmed this session) — the script has not been executed at its real target
scale (50/s for 10 minutes, 200 concurrent VUs). I deliberately did not attempt to
approximate that scale by hammering the local embedded-Postgres dev database directly —
that would create ~30,000 synthetic leads in a shared dev DB other tests and manual
verification runs depend on, for a result that wouldn't prove anything about the actual
target (staging infrastructure that doesn't exist yet) anyway.

**Full-scale execution requires, in order:**
1. A staging environment (Phase 6 Task 6.1 — separate Supabase/Railway/Vercel projects,
   100% owner-blocked, see `owner-actions.md`'s new Phase 6 section).
2. k6 installed wherever the load test actually runs (`brew install k6`, or a CI runner
   image that has it — trivial once decided, not owner-blocked in the account-creation
   sense, just not done yet since there's nothing to point it at).
3. A real `AUTH_BEARER_TOKEN` from a login against the staging Supabase project (staging
   runs with `AUTH_DEV_BYPASS=0` per `docs/staging-setup.md` — no shortcut here, by
   design).

## Assertions the script encodes (from the pack's own EXIT GATE wording)

- Inbox ack p95 < 500ms (`inbox_ack_duration` threshold)
- Read-model query p95 < 800ms (`read_model_duration` threshold)
- Zero event loss — deliberately NOT asserted from k6's HTTP responses alone (a 200
  only proves "the API accepted it," not "it was durably processed"). The real
  reconciliation point already exists in this codebase: any event whose processing
  goes unknown surfaces as a `reconciliation_cases` row (Phase 2's own runtime
  guarantee) — the actual zero-loss proof, once this runs for real, is "sent event
  count == created leads count, and zero new open reconciliation_cases for this run's
  events," checked by comparing `sent eventIds` (this script's own log) against the
  database after the run completes. That comparison script doesn't exist yet either —
  logged here rather than written blind against an environment that doesn't exist.
